# Working Protocol

更新时间：`2026-03-31`

## 这个目录的使用方式

新对话如果想快速接管项目，应默认先读：

1. `code_plan/README.md`
2. `code_plan/PROJECT_SNAPSHOT.md`
3. `code_plan/FEATURE_STATUS.md`
4. `code_plan/ROADMAP_NOTES.md`

读取后，再根据任务去看具体代码。

## 助手的默认协作规范

- 对涉及抓取链路、schema、部署、前端筛选、数据源接入的任务：
  - 优先读 `code_plan/`
  - 再读对应脚本

- 如果本次任务属于结构性优化：
  - 完成代码后，主动判断是否要更新 `code_plan/`

- 如果发现 `code_plan/` 与代码不一致：
  - 以代码为准
  - 在回答中指出差异
  - 如有必要，顺手更新 `code_plan/`

## 什么叫“建议更新 code_plan”

出现以下情况时，建议主动提出更新：

- 新增核心功能
- 删除或替换核心功能
- 数据源或部署方式变化
- 新增重要约束
- 新增明确的“不建议做”的路线
- 前端筛选维度变化
- 运行方式变化
- 测试入口变化

## 推荐更新方式

- 优先更新现有文件，不要盲目新增很多文档
- 只有当信息量明显增多时，再新增文件
- 保持短、准、可扫描
- 不要把临时命令输出原样灌进来

## 推荐写法

- 功能已完成：`✅`
- 功能部分完成或不稳定：`⚠️`
- 明确建议增加：`建议增加`
- 明确建议忽略：`建议忽略`

## 当前对话约定

- `code_plan/` 的目标是“让新对话快速理解当前项目”
- 它不追求完整替代 README、Git 历史或 issue 跟踪
- 只保留足以帮助接管的高价值信息
